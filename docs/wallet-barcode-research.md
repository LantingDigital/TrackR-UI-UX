# TrackR Wallet: Barcode/QR Code Research
# Researcher output — March 2026

## Recommendation: Option A (Extract + Regenerate), with original image backup

Extract the barcode data from the user's photo, regenerate a clean code from that data, store
both the raw data string and the original image URI. This is the right approach for UX, reliability,
and scanability. Details below.

---

## 1. QR Code Equivalence: Does Regenerating Work?

YES. Regenerating a QR code from extracted data always produces a valid, scannable QR code —
even if it looks visually different from the original.

Why QR codes can look different for the same data:
- The QR spec defines 8 possible "masking patterns" applied during encoding. A generator picks
  whichever mask produces the best score (balanced distribution of dark/light). Different libraries
  may choose different masks.
- Error correction level (L/M/Q/H) changes both the visual pattern and the physical size.
- The QR "version" (size, 1-40) is chosen based on data length and error correction. Same data +
  same ECC level always picks the same version, but masking varies.

Practical result: a QR code generated by react-native-qrcode-svg encoding the same string as
the original park pass QR code will scan identically at every gate. The pixel arrangement does
not matter to scanners — only the encoded data matters.

---

## 2. Barcode Equivalence: 1D (Code 128, Code 39) and 2D (PDF417)

YES, with one important caveat: the checksum must be recalculated correctly.

### Code 128 (most common on theme park physical tickets)
- 106 printable patterns, 3 start character sets (A, B, C)
- Mandatory modulo-103 check digit appended at generation
- Different libraries may choose character set A vs B for the same ASCII data — visually different
  bar widths, but decode to identical data
- A library-regenerated Code 128 from extracted digits will scan perfectly at any gate

### Code 39
- Simpler, no mandatory checksum (optional mod-43 check)
- Easiest to regenerate — purely deterministic for the same input string

### PDF417 (used on some paper tickets and season pass cards)
- 2D stacked barcode, Reed-Solomon error correction built in
- Holds up to 1,850 alphanumeric chars or 1,108 bytes of binary
- Regeneration from extracted data: works cleanly as long as the compaction mode (text/numeric/byte)
  is set to match what the library auto-detects from the data
- Most theme park PDF417 codes are text or numeric — no issues expected

Key caveat: some parks encode additional invisible metadata or structured data in their barcodes
(e.g., GS1 data structures, AIM symbology identifiers, specific encoding flags). If the scan
extracts raw decoded text and you regenerate from that, any extra invisible wrapper is gone.
In practice, major parks (Disney, Universal, Six Flags) use standard encodings that decode and
re-encode cleanly.

## 3. Recommended Approach: Option C (Hybrid)

Store both:
1. The extracted data string (source of truth for regeneration)
2. The original image URI (local device storage / Firebase Storage as fallback)

Generate the display code from the extracted string at render time.

Why hybrid over pure Option A:
- If extraction fails or gets partial data, the original image is the safety net
- For passes that encode binary blobs rather than readable text, you may not be able to
  regenerate — having the original image as fallback covers this
- Storage cost is negligible (the raw string is tiny; the original image is already on-device)

## 4. React Native / Expo SDK 54 Library Recommendations

### Scanning (input)

**Primary: expo-camera (Camera.scanFromURLAsync)**
- expo-barcode-scanner was deprecated in SDK 50, removed in SDK 52 — do not use
- expo-camera is the official replacement and is already in TrackR's stack
- Camera.scanFromURLAsync(uri, [barcode types]) scans from an image file URI, not a live camera
- Supported types: qr, code128, code39, pdf417, aztec, ean13, ean8, datamatrix, upc_a, upc_e
- iOS limitation: expo-camera only scans QR codes from images on iOS (not 1D barcodes from images)
- Android: full support for all types from image URIs

**Cross-platform fallback for 1D barcodes from images: react-native-vision-camera**
- Uses MLKit (Android) and Vision Kit (iOS) — full barcode type support from image files
- Expo compatible with Development Build (not Expo Go)
- Heavier dependency but gold standard for scanning from image files on both platforms
- Repo: https://github.com/mrousavy/react-native-vision-camera

Practical recommendation: Use expo-camera for QR codes (covers Disney MagicMobile, Universal
app passes) and test it for Code 128/PDF417. If iOS image scanning of 1D barcodes is needed,
add vision-camera. Most users will photograph a QR code so expo-camera alone may be sufficient.

### Generation (display at gate)

**QR codes: react-native-qrcode-svg**
- npm: react-native-qrcode-svg
- SVG-based, scales perfectly at any screen size — critical for gate scanners
- Requires react-native-svg (already common in Expo projects)
- Simple: <QRCode value="your-extracted-string" size={250} />
- Maintained by Expensify, actively updated

**Code 128 + PDF417: react-native-barcode-creator**
- npm: react-native-barcode-creator
- Supports: EAN13, Code128, PDF417, AZTEC, UPCA, QR
- Native generation — requires Development Build (not Expo Go), use npx expo prebuild
- Expo install: npx expo install react-native-barcode-creator
- Repo: https://github.com/VittoriDavide/react-native-barcode-creator

**Alternative for PDF417 only: expo-barcode-pdf417**
- Forked to replace ART dependency with react-native-svg — Expo-friendly
- Lighter if you only need PDF417
- Repo: https://github.com/notpushkin/expo-barcode-pdf417

### Image picking (user flow entry point)

- expo-image-picker — already a standard Expo package
- launchImageLibraryAsync() to let users pick a screenshot from their camera roll
- OR a camera capture flow with expo-camera directly

## 5. Legal Considerations

Storing a personal copy of a barcode you own for your own use falls into a legal gray area
that is practically very low risk. Key findings:

- Disney, Universal, Six Flags, and Cedar Fair all prohibit TRANSFER of passes to other people
- None of the reviewed terms specifically prohibit storing your own barcode digitally for
  personal use at the gate (you'd be presenting your own pass, just in a different app)
- Apple Wallet and Google Wallet already do exactly this — store and display copies of passes —
  and are used by these parks officially (Disney MagicMobile, Universal app, etc.)
- TrackR is not facilitating transfer or resale — it's a personal pass wallet, same as Apple
  Wallet but for enthusiasts who want coaster tracking alongside their passes
- Risk is essentially the same as screenshotting your own pass to show at the gate — which is
  already common practice and not something parks actively prevent

Add a ToS note in TrackR: "This wallet stores your passes for personal use only. Sharing or
transferring access to another person's pass violates park terms of service."

## 6. Recommended User Flow

1. User taps "Add Pass" in the wallet screen
2. Choose: "Take Photo" (camera) or "Import from Photos" (image picker)
3. expo-camera scans the image for all supported barcode types
4. If a barcode is found:
   a. Display the extracted data string for confirmation ("Found: QR Code — 1234567890")
   b. User names the pass (e.g., "Disney Annual Pass", "Cedar Point Platinum")
   c. App stores: { type, data, parkId (optional), label, originalImageUri }
   d. App generates a clean display code from the data at render time
5. If no barcode found:
   a. Offer "Save as image only" fallback — store the cleaned-up photo without data extraction
   b. This covers edge cases (laminated cards with glare, obscured codes, etc.)

At the gate:
- User opens pass from wallet
- Screen shows the regenerated QR/barcode at full brightness, max size
- Auto-brightness boost on open (common Apple Wallet behavior)
- Optional: lock screen rotation to portrait while pass is displayed

## 7. Data Schema for Wallet (Firestore)

Suggested addition to the existing schema:

```typescript
// users/{userId}/passes/{passId}
interface WalletPass {
  id: string;
  label: string;              // "Disney Annual Pass - Caleb"
  parkId: string | null;      // FK to parks collection if known
  barcodeType: 'qr' | 'code128' | 'code39' | 'pdf417' | 'aztec' | 'image-only';
  barcodeData: string | null; // The extracted string/number — null if image-only
  originalImageUri: string;   // Firebase Storage URL for the original photo
  displayColor: string | null; // Optional tint for the card
  createdAt: Timestamp;
  lastUsedAt: Timestamp | null;
  sortOrder: number;
}
```

Store originalImageUri in Firebase Storage at: passes/{userId}/{passId}/original.jpg
